This POC is really comparing three very different PDF strategies, even though all three generate the same outputs in main.go.

chromedp in chromedp/chromedp.go is “HTML/CSS rendered by headless Chrome, then printed to PDF.”
maroto in maroto/ma-invoice.go, ma-badge.go, and ma-agreement.go is a Go-native layout DSL for rows, columns, text, images, and pages.
pdfcpu in this repo is not being used as a high-level layout engine; the code in pdfcpu/pc-invoice.go, pc-badge.go, and pc-agreement.go manually writes raw PDF drawing commands and then uses pdfcpu mainly to optimize/normalize the generated PDF.

Differences

chromedp

Best when the source is already HTML/CSS or needs browser-quality styling.
Strongest visual fidelity for templates, branding, tables, print CSS, and rich layouts.
Operationally heaviest for a server because you must deploy Chrome/Chromium and manage browser startup, memory, and timeouts.
In this POC it also depends on a live external QR image URL by default via https://api.qrserver.com/... in chromedp/chromedp.go, which is a server deployment risk unless you switch to local QR generation.
maroto

Best when PDFs are structured business documents built in Go: invoices, statements, receipts, agreements, certificates.
Easier to maintain than low-level PDF drawing because layout is expressed as rows/cols/components.
Much lighter for servers than chromedp; no browser dependency.
Good balance of control and maintainability, but less flexible than HTML/CSS for pixel-perfect design-heavy documents.
In this repo it even embeds fonts for portability in maroto/ma-agreement.go, which is a plus for deployment.
pdfcpu

Best when you need low-level PDF control or PDF post-processing, not when you want a pleasant templating workflow.
Highest engineering effort for document authoring because every coordinate, wrap rule, font limitation, and drawing primitive is manual.
Lightest runtime footprint on a server because there is no browser, but highest development and maintenance cost.
Good for stamping, merging, optimizing, validating, watermarking, or specialized PDF manipulation.
In this POC, it is the most custom and brittle path: manual word wrap, manual encoding, manual page streams, manual QR embedding.
Best use cases in this project

Invoice

chromedp: best if your invoice already exists as a web template or needs exact branding.
maroto: best if the invoice structure is stable and generated fully from backend data.
pdfcpu: only if you need ultra-custom low-level output or want to avoid any higher-level layout layer.
Badge / payment card

chromedp: good for marketing-style badge layouts, but current QR loading is network-dependent.
maroto: very suitable here; it already generates the QR locally and the layout is simple.
pdfcpu: also suitable for compact fixed-layout cards, but harder to evolve.
Agreement / multi-page forms

chromedp: best for long legal text if you want readable HTML templates and easier copy updates.
maroto: good if the agreement structure is fixed and tabular/form-like.
pdfcpu: workable, but the hardest to maintain as clauses and pagination change.
For server deployment

Choose chromedp if:
You want the fastest path from HTML template to polished PDF.
Your team already knows HTML/CSS better than PDF internals.
You are okay shipping Chrome/Chromium in Docker/VMs and sizing for higher CPU/RAM.
Choose maroto if:
You want a backend-only Go service with simpler operations.
Your PDFs are mostly structured documents, not brochure-level web layouts.
You want the best maintainability/runtime tradeoff.
Choose pdfcpu if:
Your primary need is PDF manipulation or very controlled low-level generation.
You can afford the extra development complexity.
You do not want a browser runtime at all.
Practical recommendation
For a server-hosted production service, I’d rank them:

maroto for most transactional PDFs.
chromedp for high-fidelity HTML-driven templates.
pdfcpu as a specialist tool for low-level generation or post-processing, not the default authoring path.
So for this POC:

Use maroto as the default backend PDF engine.
Use chromedp only for templates that truly benefit from HTML/CSS rendering.
Keep pdfcpu for advanced PDF manipulation or edge-case rendering where low-level control matters.
